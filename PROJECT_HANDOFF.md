## 🎯 项目目标与背景

### 核心目标
将 CustomNPC-Plus (1.7.10) 的高优先级特性回移到 CustomNPCs 1.6.4，同时保持：
- ✅ Java 7 兼容性（硬上限，ASM 4.1 限制）
- ✅ 增量式回移策略（行为语义优先，避免大规模重构）
- ✅ 原版 1.6.4 API（无 Mixin，用自有渲染器）

### 为什么回移到 1.6.4？
- **目标用户群**：仍在维护 1.6.4 服务器的社区（如中国区老牌服务器）
- **技术优势**：1.6.4 稳定性高，mod 生态成熟，性能优化充分
- **避免版本跳跃**：1.7.10 → 1.12.2 跨度大，不如先稳固 1.6.4 基础

## 🛠️ 技术决策记录（ADR）- 必读

### ADR 1：编译目标锁定 Java 7（硬上限）
**背景**：1.6.4 FML 内置 ASM 4.1，只能解析 class 版本 ≤51（Java 7）  
**决策**：永远不升级到 Java 8，已实测 Java 8 字节码导致 mod 被 FML 丢弃  
**影响**：
- ✅ 可用语法：diamond、try-with-resources、multi-catch、switch-on-String
- ❌ 禁用语法：Lambda、Stream、Optional、default method

### ADR 2：无 Mixin，用自有渲染器
**背景**：1.6.4 无 Mixin 支持，但 NPC 渲染器是我们自己的代码  
**决策**：直接修改 `RenderNPCInterface` 等自有渲染器，无需 coremod  
**影响**：碰撞箱缩放、色调系统、皮肤覆盖层均已成功实现

### ADR 3：脚本引擎用纯 JSR-223
**背景**：CNPC+ 硬依赖 Nashorn（JDK 8+），与 Java 7 冲突  
**决策**：用 `javax.script` 标准 API，Java 7 自动用 Rhino、Java 8 自动用 Nashorn  
**影响**：脚本功能可兼容两个运行时

## 🤖 子代理调度策略

### 何时使用子代理
- **独立功能实现**：如自然生成 GUI、对话视觉取色器
- **多文件修复**：如当前的 Display 布局修复（涉及 3 个文件）
- **调查型任务**：如"查找 CNPC+ 的某特性实现方式"

### 何时自己动手
- **单文件小改动**：如修改语言键、调整按钮位置
- **决策型任务**：如选择下一个优先级任务、评估技术方案
- **编排型任务**：如创建子代理、分配任务、审查代码

### 子代理最佳实践
1. **明确任务边界**：一个子代理只做一个独立功能
2. **提供详细 prompt**：包含技术方案、文件路径、测试步骤
3. **交接文档先行**：创建 HANDOFF.md 或类似文档供子代理参考
4. **验证再合并**：子代理完成后必须编译验证+人工测试
